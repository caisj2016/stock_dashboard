# 代码准则

本项目的目标架构很明确：

- Python 只做数据抓取和预处理
- Java Spring Boot 做主业务逻辑和统一 API
- 前端只做展示和交互
- 不再引回 Flask 作为业务后端

这份准则用于约束后续改动，避免代码重新长回“前后端职责混杂”的状态。

## 1. 架构边界

### Python

Python 代码只允许放在 `python_fetchers/` 及其后续扩展模块中，职责仅限于：

- 外部数据源抓取
- HTML / RSS / JSON 解析
- 原始字段清洗
- 基础预处理和标准化

Python 不应承担：

- 页面路由
- API 对外接口
- 页面级数据聚合
- 筛选规则和业务判断
- DTO 组装

如果一个逻辑需要回答“页面应该显示什么”或“某个信号是否成立”，它应放在 Java，不应放在 Python。

### Java Spring Boot

Java 是系统主后端，负责：

- 业务规则
- 页面级数据聚合
- 统一 API
- DTO 设计
- 缓存、容错、错误处理
- 与 Python 抓取结果或外部数据源对接

分层要求：

- `controller` 只接收请求、返回响应，不写业务逻辑
- `service` / `service/impl` 承担业务编排和规则判断
- `client` 负责外部源访问
- `repository` 负责持久化
- `dto` 只负责接口输入输出

### 前端

前端只负责：

- 请求 Java API
- 页面渲染
- loading / empty / error 状态
- 用户交互
- 少量纯展示格式化

前端不应承担：

- 技术指标计算
- 选股规则判断
- ownership / digest / insight 业务推导
- 多数据源聚合
- Java 返回结构之外的业务补丁逻辑

如果前端出现“根据多个字段再推导业务结论”的代码，应优先回收至 Java。

## 2. 改代码时的优先原则

后续改动默认遵循以下顺序：

1. 先改 Java DTO 和 API，让页面直接可用
2. 再简化前端消费方式
3. 最后再考虑是否需要补 Python 抓取预处理

不要反过来做：

- 不先让前端兜业务
- 不先在 Python 拼页面结果
- 不为了快而把业务判断塞进 `utils.js`

## 3. Java 代码规则

### Controller

- 每个接口只做参数接收、调用 service、返回 `ApiResponse`
- 不在 Controller 中写抓取、计算、排序、筛选逻辑
- 新接口必须补 Swagger 注解

### Service

- 业务逻辑集中在 `service/impl`
- 能返回页面直接使用的数据，就不要把二次拼装留给前端
- 同一页面需要的聚合数据，优先在一个 service 中完成

### DTO

- DTO 以页面消费为导向，不以底层数据源字段为导向
- 字段命名保持稳定，优先使用 `camelCase`
- 新增 DTO 时补 `@Schema`
- 修改 DTO 字段前，先检查前端调用点

### 异常和响应

- 统一使用 `ApiResponse`
- 错误码要稳定、可读
- 不向前端透传难以理解的上游异常文本，必要时做语义化包装

## 4. Python 代码规则

- Python 模块应尽量无状态、可复用
- 抓取逻辑与清洗逻辑尽量放在同一 fetcher 模块内
- 不新增页面耦合字段，如 `cardClass`、`badgeTone`、`shouldHighlight`
- Python 输出优先是“原始但规整”的中间结果，不是页面最终文案

如果某个 Python 函数开始出现以下特征，应视为越界：

- 直接生成页面摘要文案
- 直接决定 UI 状态
- 带有明显页面命名语义
- 依赖某个具体页面的展示结构

## 5. 前端代码规则

### API 调用

- 默认使用 Java API
- 不新增兼容旧后端的请求分支
- `utils.js` 不再继续扩展业务兼容层

### 展示逻辑

- 可以做格式化，如百分比、颜色、日期显示
- 可以做交互控制，如 tab、筛选按钮、展开收起
- 不做业务推导，如信号判定、策略评分、摘要生成

### 文件职责

- `utils.js` 只放通用请求和轻量工具
- 页面脚本只处理该页面的渲染与交互
- 跨页面复用逻辑优先抽成小工具，不做“大杂烩兼容文件”

## 6. 变更策略

项目默认采用小步改造：

- 先改最小闭环
- 保持现有功能尽量不坏
- 不一口气大改多个层级
- 每次改动都尽量能独立验证

优先改：

- 高收益、低风险、职责不清的边界点
- DTO 与前端契约不清楚的地方
- 仍然残留业务逻辑的前端代码

谨慎改：

- 外部数据源抓取
- 已经稳定运行的页面主链路
- 一次牵动前端、Java、Python 三层的改动

## 7. 提交前检查

提交前至少确认：

- 这个逻辑是不是放在了正确层
- Controller 里有没有混入业务逻辑
- 前端有没有新增业务判断
- Python 有没有输出页面语义字段
- Swagger 是否需要同步更新
- DTO 变更是否影响现有页面

## 8. 测试与验证

Java 改动后至少执行：

```powershell
cd backend
mvn test
```

前端相关改动应至少确认：

- 页面能正常加载
- API 返回字段与前端消费一致
- 不依赖已废弃后端路径

如果改动涉及新的抓取或解析逻辑，还应补最小可验证样例或单测。

## 9. 禁止事项

以下做法默认禁止：

- 在前端新增技术指标计算
- 在 Python 新增页面级业务聚合
- 在 Controller 中直接写复杂业务逻辑
- 为了兼容旧代码继续扩大 `utils.js` 的过渡职责
- 修改接口返回结构却不检查前端调用点
- 把上游数据源字段原样透传给页面并要求前端自己解释

## 10. 一句话判断法

当你不确定代码该放哪一层时，用这句判断：

“这段代码是在解决数据源问题，还是在决定页面应该显示什么？”

- 如果是“数据源问题”，优先放 Python fetcher 或 Java client
- 如果是“页面应该显示什么”，优先放 Java service / DTO
- 如果只是“怎么显示出来”，才放前端
